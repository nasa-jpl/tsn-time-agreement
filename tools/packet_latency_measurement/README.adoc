= Packet Latency Measurement Tool

A multi-threaded C++ program for measuring packet latency between network interfaces using hardware timestamping.

== Features

* Hardware timestamping support for accurate latency measurements
* Sends and receives raw ethernet frames simultaneously using multi-threading
* Fixed EtherType: 0x88BF (configurable at compile time)
* Always uses VLAN tagging (802.1Q)
* Works on Linux using AF_PACKET raw sockets
* Custom test packet format with sequence numbers, source IDs, and timestamps
* CSV output for post-processing latency analysis
* Broadcast or unicast MAC addressing support

# Build the project

== Usage

The program requires root privileges to access raw sockets and hardware timestamping.

=== Syntax

[source,bash]
----
sudo ./packet_latency_measurement [options] <send_interface> <recv_interface>
----

=== Required Parameters

* `-s, --send-interface` - Network interface to send packets from (e.g., eth0, enp27s0)
* `-r, --recv-interface` - Network interface to receive packets on (e.g., eth1, enp30s0)
* `--source-id` - Unique source ID for this instance (used to identify packet origin)

=== Optional Parameters

* `-d, --dest-mac` - Destination MAC address (default: `ff:ff:ff:ff:ff:ff` for broadcast)
* `-v, --vlan-id` - VLAN ID (1-4095, default: 10)
* `-c, --count` - Number of packets to send (default: 10)
* `-i, --interval` - Delay between sent packets in milliseconds (default: 500)
* `-o, --output` - CSV file to write timestamp data
* `-h, --help` - Print usage information

=== Fixed Settings

* *EtherType*: 0x88BF (not configurable at runtime)
* *VLAN tagging*: Always enabled
* *Receive timeout*: 2000ms after send completion (not configurable)

=== Examples

Basic usage with source ID 1:

[source,bash]
----
sudo ./packet_latency_measurement -s eth0 -r eth1 --source-id 1
----

Or using positional arguments:

[source,bash]
----
sudo ./packet_latency_measurement eth0 eth1 --source-id 1
----

With VLAN ID 10, 20 packets, 3 second interval, and CSV output:

[source,bash]
----
sudo ./packet_latency_measurement -s enp27s0 -r enp30s0 --source-id 1 -v 10 -c 20 -i 3000 -o timestamps.csv
----

Bidirectional latency measurement (two instances on same interface):

[source,bash]
----
# Instance 1 (terminal 1) - sends and receives on enp27s0
sudo ./packet_latency_measurement -s enp27s0 -r enp27s0 --source-id 1 -o instance1.csv

# Instance 2 (terminal 2) - sends and receives on enp30s0
sudo ./packet_latency_measurement -s enp30s0 -r enp30s0 --source-id 2 -o instance2.csv
----

== How It Works

1. The receiver thread starts first and binds to the receive interface
2. After a 500ms startup delay, the sender thread starts sending packets
3. Both threads run concurrently:
   * *Sender*: Constructs VLAN-tagged ethernet frames with EtherType 0x88BF, sends them at the specified interval, and captures TX hardware timestamps
   * *Receiver*: Listens for incoming VLAN-tagged frames with EtherType 0x88BF and captures RX hardware timestamps
4. The receiver runs for a calculated time period: `500ms + (count × interval_ms) + 2000ms`
   * This allows it to catch all sent packets plus any duplicates or delayed packets
   * The program exits when the receiver's time expires
5. All timestamps (TX and RX) are written to a CSV file for post-processing

=== Hardware Timestamping

The tool enables hardware timestamping on both send and receive interfaces:

* *TX timestamps*: Retrieved from the socket error queue after packet transmission
* *RX timestamps*: Retrieved from ancillary data in `recvmsg()` control messages
* Falls back to software timestamps if hardware timestamping is unavailable

=== Handling Packet Duplicates

The receiver will catch all packets that arrive, including duplicates. Since it runs for a calculated time period (rather than stopping after a fixed count), it will report every packet received, making it ideal for testing networks that duplicate packets.

== CSV Output Format

When the `-o` option is specified, the tool writes all TX and RX timestamps to a CSV file with the following columns:

[source,csv]
----
direction,interface,sequence_number,source_id,timestamp_sec,timestamp_nsec,type
TX,enp27s0,0,1,1777066967,411510506,HW
TX,enp27s0,1,1,1777066970,412064063,HW
RX,enp27s0,0,2,1777066969,502518046,HW
RX,enp27s0,1,2,1777066972,503022145,HW
----

*Columns:*

* `direction` - TX (transmitted) or RX (received)
* `interface` - Interface name (send interface for TX, receive interface for RX)
* `sequence_number` - Packet sequence number (0, 1, 2, ...)
* `source_id` - Source ID from the packet payload (identifies which instance sent it)
* `timestamp_sec` - Timestamp seconds (Unix epoch)
* `timestamp_nsec` - Timestamp nanoseconds (zero-padded to 9 digits)
* `type` - HW (hardware timestamp) or SW (software timestamp)

=== Post-Processing for Latency Measurement

To calculate bidirectional latency between two interfaces, run two instances and match timestamps:

1. Instance 1 TX (src=1) → Instance 2 RX (src=1) = Forward latency
2. Instance 2 TX (src=2) → Instance 1 RX (src=2) = Reverse latency

Match rows by `sequence_number` and `source_id`.

== Packet Format

All packets are VLAN-tagged (802.1Q):

* Ethernet header with VLAN tag (18 bytes)
** Destination MAC (6 bytes)
** Source MAC (6 bytes)
** VLAN Tag (4 bytes)
*** TPID: 0x8100 (2 bytes) - Tag Protocol Identifier
*** TCI: VLAN ID (2 bytes) - Tag Control Information (12-bit VID + 3-bit PCP + 1-bit DEI)
** EtherType: 0x88BF (2 bytes)
* Sequence number (4 bytes)
* Source ID (4 bytes)
* Timestamp (8 bytes)
* Payload message (60 bytes)

*Total size: 94 bytes*

== Platform Notes

* *Linux only*: Uses AF_PACKET raw sockets
** Requires root privileges for raw socket access and hardware timestamping
** Uses `SO_TIMESTAMPING` for hardware timestamp capture
** Uses standard `SO_RCVTIMEO` socket option for read timeouts (100ms for responsive checking)
** Uses `recvmsg()` with `PACKET_AUXDATA` to access VLAN information stripped by the kernel
** Handles both kernel-stripped and in-packet VLAN tags transparently
** BPF filter attached to prevent receiving own packets (filters by source MAC)

=== VLAN Tag Handling

The Linux kernel typically strips VLAN tags from incoming packets and provides them via auxiliary data. This tool:

* Enables `PACKET_AUXDATA` on the receive socket
* Uses `recvmsg()` to receive both packet data and control messages
* Extracts VLAN TCI (Tag Control Information) and TPID from `tpacket_auxdata` structure
* Reports whether VLAN was kernel-stripped or remained in the packet
* Works correctly regardless of kernel VLAN offload settings

=== Hardware Timestamping Requirements

Hardware timestamping requires:

* NIC driver support for hardware timestamps (most modern NICs support this)
* The tool enables `HWTSTAMP_TX_ON` and `HWTSTAMP_FILTER_ALL` on the interface
* TX timestamps are retrieved from the socket error queue using `MSG_ERRQUEUE`
* RX timestamps are retrieved from `SO_TIMESTAMPING` control messages
* If hardware timestamps are unavailable, software timestamps are used as fallback
